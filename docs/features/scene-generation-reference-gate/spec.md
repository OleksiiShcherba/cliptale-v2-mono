# Spec — scene-generation-reference-gate

> **Glossary:** [CONTEXT](./CONTEXT.md)
> **Reference module / docs / channels used:** `docs/architecture-map.md`; the predecessor feature [storyboard-reference-flows](../storyboard-reference-flows/) (spec + CONTEXT — the Star-gate / Reference-boundary contract this feature revises); the current illustration service `apps/api/src/services/storyboardIllustration.service.ts` and the rolling-window worker `apps/media-worker/src/jobs/ai-generate.referenceWindow.ts` (read for the existing gate + principal-image behaviour this feature replaces).

## 1. Context

¶1 **What we're solving.** A Creator building a storyboard draft expects every generated scene to look consistent — the same character and the same environment across scenes. Today scene generation is gated on *starring* (every reference block needs ≥1 starred result) and additionally consumes one up-front **principal image** as the global reference. Starring is a curation chore that does not actually guarantee a reference *exists yet*, and the single principal image cannot keep multiple characters/environments consistent across scenes. The result is scenes that drift visually and a confusing two-track readiness model (star here, approve a principal image there).

¶2 **Why now.** The per-cast reference flows (characters + environments, generated by a rolling window) shipped in `storyboard-reference-flows` (merged 2026-06-07). With real per-cast reference outputs now available, the up-front principal image is redundant and the starring gate is the wrong precondition: the genuine readiness condition is "every cast reference has finished generating." Aligning the gate to generation-completion and feeding scenes the per-cast outputs is the natural follow-on, and removing the principal-image track retires dead complexity before it accrues more.

¶3 **The committed approach.** Replace the starring precondition with a **Reference-done gate**: full-draft scene generation may start only when every character and environment reference block is a **ready reference block** (has ≥1 completed result usable as a reference); a per-scene regeneration is gated only on the blocks linked to that scene. Scenes consume the **selected reference output** of their linked blocks (starring now only *selects which* output, it no longer gates), and the legacy **principal image** is retired from the scene-generation path entirely.

¶4 **Traceability context.** "Ready" is defined by output-existence, not by the raw `window_status` value, specifically so that manually-added reference blocks (which carry no rolling-window state) and finished-but-unstarred blocks cannot deadlock the gate or feed a scene an empty reference — a failure mode surfaced in the devil's-advocate pass and resolved here. The Reference-boundary invariant from `storyboard-reference-flows` is preserved unchanged.

## 2. Goals

- A Creator can never start scene generation while a cast reference is unfinished — and when blocked, always learns exactly which references to finish, retry, or remove.
- Every scene's preview is generated from the actual reference outputs of the characters and environments linked to that scene, improving cross-scene visual consistency over the single-principal-image baseline.
- The storyboard readiness model is single-track: "references finished generating" is the one precondition, with the legacy principal-image step removed from the Creator's path.

## 3. Non-goals

- Changing cast extraction or cast confirmation — the flow that proposes characters/environments and their scene links is unchanged.
- Changing reference-block generation itself or the rolling-window scheduler — this feature consumes its completion signal, it does not alter how references are produced.
- Adding a watchdog/reaper for reference jobs stuck mid-generation — recovering a wedged reference block is an existing platform concern tracked in §8, not delivered here.
- Music blocks/segments — out of scope; untouched.

## 4. User stories

### US-01: Start scene generation when references are ready

**As a** Creator
**I want** to start full-draft scene generation once all my cast references have finished generating
**So that** every scene is built from real, complete reference images.

### US-02: Be told which references are blocking

**As a** Creator
**I want** the start to be refused — naming the specific unfinished references — when any cast reference is not ready
**So that** I know exactly which block to finish, retry, or remove instead of guessing.

### US-03: Regenerate a single scene without waiting on unrelated references

**As a** Creator
**I want** regenerating one scene to depend only on the references linked to that scene
**So that** an unrelated unfinished reference elsewhere in the draft does not block me.

### US-04: Generate a draft that has no cast references

**As a** Creator
**I want** scene generation to still work for a draft where no cast was ever extracted
**So that** a simple storyboard without characters/environments is not blocked.

### US-05: Get consistent characters and environments per scene

**As a** Creator
**I want** each scene to use only the reference images of the characters and environments linked to it
**So that** the right cast appears in the right scenes and nothing bleeds across unlinked scenes.

### US-06: Choose which reference output feeds scenes

**As a** Creator
**I want** to pick (by starring) which output of a reference feeds scenes, with a sensible default when I pick nothing
**So that** I control the look without being forced to curate before generating.

### US-07: Skip the principal-image step

**As a** Creator
**I want** the old up-front principal-image approval to be gone
**So that** I have one straightforward readiness path instead of two.

## 5. Acceptance criteria

### AC-01 (US-01) — happy path

**Given** an authorized Creator owns a draft whose every character and environment reference block is a ready reference block
**When** the Creator starts full-draft scene generation
**Then** the system begins generating the scenes and confirms the start, with each scene drawing on the selected reference outputs of its linked reference blocks.

### AC-02 (US-02) — error / not-ready references

**Given** an authorized Creator owns a draft where at least one character or environment reference block has not finished generating (still generating, failed with no usable result, or empty)
**When** the Creator starts full-draft scene generation
**Then** the system refuses the start when attempted (the start is rejected, not merely hidden), names each blocking reference block, and tells the Creator to finish, retry, or remove those blocks — using the existing reference-flow controls — before starting. (No new retry/force-fail affordance is introduced by this feature; "retry" and "remove" refer to the already-available re-run-flow and delete-block actions.)

### AC-03 (US-03) — per-scene scope (happy)

**Given** an authorized Creator owns a draft in which the reference blocks linked to scene S are all ready, while some unlinked reference block elsewhere is still unfinished
**When** the Creator regenerates scene S
**Then** the system starts generating scene S, ignoring the unfinished unlinked block.

### AC-03b (US-03) — per-scene scope (blocked)

**Given** an authorized Creator owns a draft in which a reference block linked to scene S has not finished generating
**When** the Creator regenerates scene S
**Then** the system refuses, naming the unfinished reference block(s) linked to that scene.

### AC-04 (US-04) — zero references allowed

**Given** an authorized Creator owns a draft that has no reference blocks at all
**When** the Creator starts scene generation
**Then** the system starts generating the scenes from their prompts and derived style description, without requiring any reference.

### AC-04b (US-05) — referenced draft: every scene must have a linked reference

**Given** an authorized Creator owns a draft that has at least one reference block, where every reference block is ready, but at least one scene has no reference block linked to it
**When** the Creator starts full-draft scene generation
**Then** the system refuses to start and names the scene(s) that have no linked reference block, telling the Creator to link a reference before starting. (The prompt-and-style-only path of AC-04 applies only to a draft that has zero reference blocks; once a draft holds any reference block, every scene must be linked.)

### AC-05 (US-05) — domain invariant: Reference boundary

**Given** an authorized Creator owns a draft where scene S is linked to some reference blocks and not linked to others, all ready
**When** scene S is generated
**Then** only the selected reference outputs of the blocks linked to S feed S, and the images of unlinked blocks are never used for S — the Reference-boundary invariant.

### AC-06 (US-06) — selected output with no explicit pick

**Given** an authorized Creator owns a draft with a ready reference block linked to scene S that has no starred result
**When** scene S is generated
**Then** the system feeds S the deterministic default — the latest completed output of that block — so the scene is never left with an empty reference.

### AC-06b (US-06) — selected output honours the star

**Given** an authorized Creator owns a draft with a ready reference block linked to scene S whose primary starred result is set
**When** scene S is generated
**Then** the system feeds S that primary starred output for that block — provided it is a completed usable output; if the primary starred result is not a completed usable output (e.g. it points at a deleted or still-generating result), selection falls back to the deterministic default (latest completed output), never to an empty reference.

### AC-07 (US-01) — cross-context readiness

**Given** an authorized Creator's draft where a character reference block's first generation is still in progress in the reference-generation context (the media-worker rolling-window), so the block has no persisted completed output yet
**When** the Creator starts full-draft scene generation
**Then** the system blocks the start because the gate reads no persisted completed output for that block (output-existence read — not a live event/await), and reflects it as a blocking block — scene generation cannot outrun reference generation. (This is the still-generating instance of AC-02; readiness is always the persisted-state read, never a separate completion-event subscription.)

### AC-08 (US-07) — principal image retired

**Given** an authorized Creator owns a draft (including one that previously held a legacy principal-image record)
**When** the Creator starts scene generation
**Then** the system neither generates nor requires any principal image, and offers no principal-image approval step; readiness is decided solely by the Reference-done gate. Any pre-existing legacy principal-image record is **ignored on read** at runtime (it never feeds a scene and never affects the gate); row-level migration of those records (drop / backfill) is deferred to `data-model` (§8) and is not required for this behaviour.

### AC-09 (US-01) — authorization

**Given** a Creator who does not own a given draft
**When** they attempt to start scene generation for it
**Then** the system denies the action and does not reveal the draft's reference or scene state.

## 6.1 Security / privacy

- **Data classification:** internal — storyboard drafts, reference blocks, and generated images are private to the owning Creator.
- **Personal data touched:** none new — this feature changes a precondition and a reference-selection path; it introduces no new personal-data fields.
- **AuthZ/AuthN impact:** no new permission boundary. Every start/regenerate/readiness operation continues to resolve the draft through the caller's ownership (the existing owner-scoped draft resolution); a non-owner is denied and the draft's existence/state is not disclosed.
- **Abuse cases:**
  - Cross-tenant start (a Creator starting generation on a draft they do not own): deny via owner-scoped resolution; do not reveal the draft's reference/scene state.
  - Gate-bypass via per-scene start (regenerating scenes one-by-one to dodge the full-set gate): acceptable by design — each per-scene start independently enforces the scene-linked readiness scope, so no scene is generated against an unfinished linked reference.
  - Reference-state leak through the blocking-block error (the rejection naming references on a draft the caller can't access): prevented by resolving ownership before any gate evaluation, so non-owners never reach the naming path.
- **Security review:** N/A — no new authz boundary, no new PII, no new external surface; behaviour stays within the existing owner-scoped storyboard ACL.

- **Reference-utilization rate** (share of generated scenes that consumed ≥1 linked reference output, among scenes that have ≥1 linked reference block) — baseline: TBD (measure first week post-rollout via generation telemetry), target: ≥ 80% within 30 days.
