## Findings

### F1 (Minor, stage-1 — AC-02/AC-10) — no single chained regression asserting non-empty `scene_ids` after a real plan completion

`apps/api/src/__tests__/integration/storyboardPipeline.e2e.test.ts:452-468,485-504` seeds `scene_ids` / `scene_block_ids` directly into the cast proposal, so it would still pass if `materializeScenePlan` were removed — it does not drive the real `materialize → getScenes → prune` chain that r6 explicitly asked for. The seam **is** individually well-covered (materialize writes real scene blocks — `materializeScenePlan.integration.test.ts:114-133`; cast-extract keeps ids when `getScenes` returns scenes and prunes all when empty — `cast-extract.job.test.ts:159-184,190-201`; order markCompleted<materialize<advance — `storyboardPlan.job.test.ts:834-865`), which is why this is Minor not Blocker. But the one chained regression r6 named is not assembled.

**Resolution: Fix now.** Add one integration test: call `materializeScenePlanBlocks` then `processCastExtractJob` with the real `getScenes`, and assert the stored proposal's `scene_block_ids` are non-empty UUIDs matching real `storyboard_blocks.id`.

### F2 (Minor, stage-2) — comment overstates recovery on materialize failure

`apps/media-worker/src/jobs/storyboardPlan.job.ts:323-325`: on materialize failure the plan job swallows the error and returns success, so BullMQ does **not** retry — only the 10-minute reaper (`storyboardPipelineReaper.job.ts:51-101`) recovers the stuck `scene` phase. The "stays running for retry/reaper" wording implies a retry path that does not exist. Behavior is safe (no race), but misleading and a slow recovery (≤10 min) for a transient DB blip.

**Resolution: Fix now.** Rethrow on materialize failure so BullMQ retries the plan job (`markRunning`/`markCompleted` are idempotent; the plan_job row being `completed` re-runs cleanly), letting materialization succeed on attempt 2 well before the 10-min reaper bound. Fix the comment to match.

### F3 (Minor, stage-2, pre-existing) — re-materialize does not clear `storyboard_reference_blocks`

`apps/media-worker/src/jobs/materializeScenePlan.ts:64-66`: on a regenerate that re-mints scene-block ids, prior reference blocks (and their stored `scene_block_ids`) point at deleted scene ids. This exactly mirrors the api `applyLatestCompletedPlan` / `replaceStoryboard` behavior (`storyboard.repository.ts:233-242`), so it is **not** a regression from this fix; reference-block lifecycle is owned elsewhere.

**Resolution: Not an issue** (this round). Pre-existing, consistent with the api path, owned outside the pipeline materializer. Flagged for conscious record only; no action.
