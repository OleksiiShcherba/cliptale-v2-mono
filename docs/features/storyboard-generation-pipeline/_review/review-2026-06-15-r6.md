## Finding

### F1 (Blocker, stage-1 — AC-02, consequentially AC-10/AC-01) — cast-extraction races ahead of scene-block materialization → "0 scenes" on every reference

**Symptom:** `ReviewCastProposalModal` renders `N scenes` from `ref.scene_ids.length` (`apps/web-editor/src/features/storyboard/components/ReviewCastProposalModal.tsx:106-108`). Every reference shows `0` on every pipeline run.

**Root cause — ordering, traced file:line:**
1. Scene blocks (`storyboard_blocks` `block_type='scene'`) are INSERTed **only** by `applyLatestCompletedPlan` (`apps/api/src/services/storyboardPlanApply.service.ts:186-207`), reachable solely via the client HTTP endpoint `POST /storyboards/:draftId/apply-latest-plan` (`apps/api/src/controllers/storyboard.controller.ts:70`, `routes/storyboard.routes.ts:40`). The worker scene-plan job persists only the plan JSON (`apps/media-worker/src/jobs/storyboardPlan.job.ts:285-290`); it never creates scene blocks.
2. On scene-plan completion the pipeline advances `scene→completed, reference_data→running` and **immediately** enqueues cast-extract (`storyboardPlan.job.ts:298-305` → `onSceneGenerationComplete` `apps/media-worker/src/jobs/storyboardPipelineHooks.ts:283-311` → `enqueueCastExtraction.ts:49`).
3. `onSceneGenerationComplete` guards **only** the status state-machine (`canTransition` on `scene_status`/`reference_data_status`, `storyboardPipelineHooks.ts:292-293`) — it does **not** check scene-block existence before advancing/enqueuing.
4. Cast-extract reads valid scene ids from `getScenes` (`SELECT … WHERE block_type='scene'`, `workerRepositories.ts:412-430`); with no blocks yet, `validSceneIds` is empty (`cast-extract.job.ts:350-352`) and `pruneSceneIds` drops every `scene_block_ids` entry (`cast-extract.job.ts:220-225,370-372`). The code comment already names the case: *"When the draft has no scenes yet (extraction raced the plan), NO id can be valid, so everything is pruned"* (`cast-extract.job.ts:217-219`).
5. The cast modal payload is projected from the (already-pruned) stored proposal (`projectCastProposalPayload` `storyboardPipelineHooks.ts:160-180`) → `scene_ids: []` for every reference.

**Why introduced by this feature:** the old flow triggered cast-extraction **after** the user applied the plan (blocks existed, ids resolved). The pipeline (T15, `1abe2af`) replaced the user-initiated client apply with a backend auto-chain on scene-plan completion, but the chain advances `reference_data` and enqueues cast-extract without ensuring the plan is materialized first. The live screenshot confirms blocks *do* get applied (toast + canvas), just **later** than cast-extract runs — i.e. a genuine ordering race, not a permanent absence.

**Blast radius beyond AC-02:** empty `scene_block_ids` ⇒ confirm-cast seeds no reference→scene links ⇒ **AC-10** (Ready refs feed scenes) is never exercised and scene-image falls to text-only for everything; `countSceneBlocks`/`listSceneBlocks` (`storyboardPipeline.trigger.service.ts:124-144`, `storyboardPipelineHooks.ts:113-120`) also see 0 if cast-extract is reached before apply (**AC-04/AC-15** adjacent).

**Fix direction (Fix now):** guarantee scene-block existence before cast-extraction. Either (a) gate the `reference_data→running` advance + `enqueueCastExtraction` on `COUNT(*) storyboard_blocks WHERE block_type='scene' > 0`, holding the phase until the plan is materialized; or (b) materialize the plan server-side in the scene-completion hook (extract the block-building from `applyLatestCompletedPlan` into a worker/transition-callable service) and only then advance. Add a regression test: scene-plan completes → cast proposal references carry non-empty `scene_ids` that match real `storyboard_blocks.id`.
