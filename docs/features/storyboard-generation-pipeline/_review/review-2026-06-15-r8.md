## Fixed this round (verified live + unit/integration green)

### F1 (Blocker, stage-1 — AC-03/AC-10) — reference blocks stuck `window_status='pending'` → reference_image phase reaped as failed
`apps/api/src/services/storyboardPipeline.confirm.service.ts:266` inserted reference blocks `'pending'` and on enqueue set only `first_job_id`, never claiming them to `'running'`. The worker's rolling-window hook `onReferenceBlockJobComplete` updates only `WHERE window_status='running'` (`ai-generate.referenceWindow.ts:108-128`), so every completed image no-op'd, blocks stayed `pending`, `onReferenceImagesAllTerminal` never fired, and the reaper failed the whole phase after 10 min. The shipped `storyboardReference.confirm.service.ts:412` claims `'running'` precisely for this reason.
**Fix:** claim `window_status='running'` alongside `first_job_id` in confirm. **Live:** reference_image now reaches `completed`. **Test:** `confirm.service.test.ts` asserts `windowStatus==='running'` post-confirm.

### F2 (Blocker, stage-1 — AC-03/AC-04) — `reference_data` stuck `awaiting_review` → scene_image offer-accept rejected by order guard
Confirming the cast claimed `reference_image` but never resolved `reference_data`; nothing in production ever set `reference_data_status='completed'`. The scene_image order guard (`prerequisitesOf` → `isPhaseResolved`) requires every earlier phase `completed|skipped`, so accepting the scene-image offer 422'd with `pipeline.phase_out_of_order`. The e2e masked it by seeding `referenceDataStatus:'completed'` (`storyboardPipeline.e2e.test.ts` AC-04 setup).
**Fix:** `claimRun` gained `alsoComplete?: PipelinePhase`; confirm passes `alsoComplete:'reference_data'` so the review resolves in the same CAS. **Live:** accept now advances scene_image→running→completed. **Test:** `confirm.service.test.ts` asserts `referenceDataStatus==='completed'`.

### F3 (Major — AC-05/AC-06/AC-07) — Creator pipeline actions don't publish realtime → live client never converges (cancel leaves the loader stuck)
Controller actions (`confirmCast`/`triggerPhase`/`cancelPhase`/`skipPhase`) only `res.json(...)`; none published `storyboard.status.updated`. `usePipelineState` learns of changes only via realtime snapshots, so after **cancel** the BlockingLoader stayed visibly stuck (server was `idle`) until a reload — the exact "trapped behind the loader" failure AC-06 exists to prevent — and the stuck overlay then blocked the corner controls (AC-15 guard click and re-trigger both missed). Skip had the same non-convergence.
**Fix:** `storyboardPipeline.controller.ts` now publishes the projected state after every Creator action via `publishStoryboardStatusUpdated` (`respondWithState`). **Live (edge2):** after the fix, cancel released the loader; AC-06, AC-15 (plain-language "Generate the scenes first…"), AC-05 resume (reload shows the running loader), and AC-07 skip (reference_data→skipped) all passed through the UI. **Test:** controller suite green (11/11).

## Also fixed this round (F5, F6 — verified live + tests)

### F5 (Major — AC-07) — corner re-trigger of `scene` / `reference_data` enqueued no worker job
`storyboardPipeline.trigger.service.ts` enqueued units only for `reference_image`/`scene_image`; for `scene`/`reference_data` it claimed the run (status→running) but enqueued nothing (the comment "the worker enqueues sub-jobs from the queue job itself" held only for the auto-start path, where `resume.service` separately calls `enqueueStoryboardPlan`). So a corner re-trigger of Scenes or References hung the phase `running` until the reaper failed it. AC-07 ("trigger any step from the corner controls") was unmet for these two phases.
**Fix:** trigger.service now enqueues a fresh `storyboard-plan` job on a `scene` re-trigger and a fresh `cast-extract` job (new job row + enqueue) on a `reference_data` re-trigger. **Test:** `trigger.service.test.ts` — two new tests assert the job is enqueued (`storyboard-plan` / `cast-extract`).

### F6 (UX — AC-07/AC-11) — skip-cast → scene images required a non-obvious intermediate trigger
After skipping the cast, `reference_image` stayed `idle`, so the scene_image order guard rejected a direct Scene-Images trigger; the Creator had to first trigger the (empty) Ref-Images corner.
**Fix:** `lifecycle.service.skipPhase` now cascades a `reference_data` skip → `reference_image` skipped (when reference_image is still idle), so Scene Images is directly triggerable text-only (AC-11). reference_image stays re-triggerable (skipped→running). **Live:** skip → direct scene_image trigger now runs (reference_data=skipped, reference_image=skipped, scene_image=running). **Test:** `lifecycle.service.test.ts` asserts the cascade.

## Open finding (deferred → spec §8)

### F4 (Major, edge race — AC-01/AC-02) — scene blocks lost after cancel → re-trigger → reload churn
In the edge run, after cancelling scene gen and re-triggering, the draft ended with **0 scene blocks** (`storyboard_blocks block_type='scene'`) despite `scene_status='completed'` and a cast proposal that had shown non-empty scenes — so reference_image/scene_image then 422 `pipeline.scenes_required`. A clean run (no cancel/reload) keeps its 5 scene blocks intact, so this is specific to the cancel→re-trigger→reload interaction (likely a stray cancelled plan job racing the re-trigger and/or the client canvas autosave clobbering server-materialized blocks). Root cause not yet isolated → deferred to spec §8 for a dedicated investigation (owner: Tech Lead).
