## Findings

### Blocker

**F1 ‚Äî Cut-over over-seeds every idle Step-2 draft as `scene running`; reintroduces the stuck-loader (regresses AC-12 / KPI ¬ß7).**
`apps/api/src/db/cutover/storyboardPipelineBackfill.ts:72` (scan predicate `OR d.status = 'step2'`) + `:159-168` (case-5 fall-through). `generation_drafts.status='step2'` is the normal status for ANY draft parked on the Step-2 screen, not a signal of in-flight scene planning. An old-flow-completed draft (status=step2, scenes already generated, no active job, no `storyboard_pipeline` row) matches the scan but none of cases 1‚Äì4, so it falls to case 5 and is seeded `scene_status='running', active_run_phase='scene'` with a fresh heartbeat ‚Äî **with no scene-plan job enqueued**. On next Step-2 open the owner is thrown behind a full-screen scene-generation loader for already-finished work; at 10 min `resume.service.ts` lazy-release marks `scene` **failed** with a fabricated "exceeded no-progress bound" message. Tenant-wide, not an edge case. The case-5 comment claims "step2 with no scenes" but neither `SCAN_SQL` nor `mapToEntry` checks scene existence. Test gap: `storyboardPipelineBackfill.integration.test.ts:120` seeds every fixture as `step2` **with** an active job, so the false-positive path is never exercised; the dry-run report carries no scene-count signal so operators can't catch it either.
**Fix:** drop the bare `d.status='step2'` predicate OR gate it on the absence of `storyboard_blocks` of `block_type='scene'`; for a `step2` draft that already has scenes and no active job, skip it (let normal lazy-create/resume handle it) or seed `scene_status='completed', active_run_phase=NULL`. Add a regression fixture: `step2` + completed scene blocks + no active job ‚Üí asserted skipped or completed, never `scene running`.

### Majors

**F2 ‚Äî AC-12: whole-phase failure of `reference_data` / `reference_image` has no "what failed + retry" surface.**
`apps/web-editor/src/features/storyboard/hooks/useStoryboardGenerationFlow.ts:102-136` maps only the `scene` and `scene_image` phases into status objects that drive the plan/illustration controls. When a reference phase fails, `active_run_phase` goes null ‚Üí `BlockingLoader` unmounts (`BlockingLoader.tsx:51`) and `state.error_message` is rendered nowhere; the only retry path is the always-on `StepCorners` buttons, which never display `error_message`. AC-12 requires telling the Creator **what failed with the option to retry** for **every** phase.
**Fix:** render a pipeline-failure banner driven by `state.error_message` + the failed phase, with a retry wired to `triggerPhase`.

**F3 ‚Äî AC-12: the scene-phase Retry / Regenerate affordances are dead no-ops.**
`useStoryboardGenerationFlow.ts:120-122,133-135` stub `retry`/`start`/`retryBlock` to `async () => null|{}`; these are consumed live at `StoryboardPageWorkspace.tsx:195` (`onRetry={() => planGeneration.retry()}`) and `:163` (`onRegenerate={illustrationGeneration.start}`). The failed-state Retry on the plan controls and the illustration Regenerate button do nothing. (Scene Regenerate via the status-block menu does work ‚Äî `StoryboardPageWorkspace.tsx:153` ‚Üí `triggerPhase`.)
**Fix:** point `onRetry`/`onRegenerate` at `triggerPhase(draftId, 'scene'|'scene_image')`, or remove the dead buttons so the only retry path is the working one.

**F4 ‚Äî AC-05 / ¬ß6 resume-freshness ‚â§2s: no re-fetch on realtime reconnect.**
`apps/web-editor/src/features/storyboard/hooks/usePipelineState.ts:41-50` passes only `onEvent`, no `onReconnect`. The realtime client fires `onReconnect` on a fresh (non-replaying) resubscribe, so any transition that occurred during a socket drop is silently missed and the held version stays stale until a full remount. The retired `useStoryboardIllustrations` wired `onReconnect: refreshFromRealtime` ‚Äî the new hook dropped that established pattern.
**Fix:** pass `onReconnect: () => getPipelineState(draftId).then(apply-if-newer)`, reusing the same `versionRef` guard.

**F5 ‚Äî AC-03 / ADR-0006 / contract: client sends `cost_estimate` in the confirm-cast body, which the contract forbids.**
`apps/web-editor/src/features/storyboard/api.ts:715-717` posts `{ cost_estimate }`; `contracts/openapi.yaml:509-519` declares `CastConfirmation` with `additionalProperties: false` and only a `references` property ‚Äî no `cost_estimate`. A strict validator 400s the confirm; the e2e (`pipeline.e2e.test.tsx:359-360`) locks the wrong shape in. The server re-validates against the persisted estimate (r2 GAP-5 fix) so there is **no under-charge**, but it is a real contract drift and it echoes the server-shown estimate back into the spend call.
**Fix:** reconcile contract and code ‚Äî either add `cost_estimate` to `CastConfirmation` in openapi (documenting the confirm-as-shown re-validation input), or drop the field from the client body (send `{}` / `{ references }` only) and update the e2e assertion.
