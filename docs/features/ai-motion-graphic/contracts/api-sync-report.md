## Flags / decisions for the Backend Lead (none blocking — all documented, none silently edited)

**Design note A — two-call authoring + browser-as-verdict (derived from §6, not invented).**
MVP1 executes graphic code **only in the browser** (SAD §1, ADR-0004/0005), so the server *cannot* compute the "runs + deterministic" verdict (AC-06/AC-09/AC-14). The §6 sequences reflect this: a stream call (`/generate`, `/refine`, SSE, non-persisting) is followed by a separate persist call (`POST /motion-graphics`, `POST /{id}/turns`) carrying the browser's `outcome`. The server stays authoritative for ownership, the ready-state invariant, the cost gate and the guardrail. This is the load-bearing interpretation of the flows — surfaced for your explicit blessing. *(Accept-as-is unless you want a different split.)*

**Drift point 1 — error envelope `{error, code?, details?}`, not `{code, message, details?}`.** Deliberate; standing team decision 2026-06-03 (same as generate-ai-flow). `error` always present; `code`/`details` additive. No action.

**Drift point 2 — error `code` emission gap + new error classes.** The current `errorHandler` attaches `code`/`details` **only** for `GateError` / `RateLimitedError`; `NotFoundError` / `ValidationError` emit bare `{error}`. Consequence for `implement`:
- `motion_graphic.estimate_revalidation_failed` → add a `GateError` subclass mirroring the existing `EstimateRevalidationFailedError` (422) — carries code+details. ✓ pattern exists.
- `motion_graphic.description_too_short`, `motion_graphic.prompt_rejected`, `motion_graphic.not_ready` → **new** `GateError` (422) subclasses so the documented codes are actually emitted.
- `motion_graphic.not_found` (404) / `motion_graphic.invalid_request` (400) → currently bare `{error}` via `NotFoundError`/`ValidationError`; the contract documents the code aspirationally (identical to the merged generate-ai-flow situation). Implement decides whether to enrich those handlers. Non-blocking.

**Drift point 3 — SSE is a new transport in `api`.** No existing `text/event-stream` endpoint in the repo (grep clean). ADR-0003 mandates SSE; `implement` introduces the SSE helper in `motionGraphicAuthoring.service.ts`. Contract documents the frame protocol (`event: token|done|error`). Not a contract bug.

**Note 4 — no DELETE endpoint.** `motion_graphics.deleted_at` (soft-delete) exists and the list filters `IS NULL`, but **no US/AC covers deletion** in MVP1, so no operation was invented (would violate "every operation maps to a user story"). The column is reserved; add the endpoint when a delete story lands.

**Note 5 — `status: generating` exposed but terminal-only persist.** The enum value is in `MotionGraphicStatus`, but the two-call model persists rows directly in `ready`/`failed`. `generating` remains the column default / forward-compat for a future server-tracked generation state.
