## Defect (stage-1, AC-04 / US-07) — attach does not survive Save

**Symptom (real client path):** On a clean storyboard block, *+ Add Media → Motion Graphic → pick a ready graphic* shows the attached preview in the picker. The picker's POST `…/media/motion-graphic` **does** persist the row in DB. But clicking **Save** (the normal next action) and reloading the storyboard leaves the block with **zero** motion-graphic media — the attachment is silently lost.

**Decisive evidence:** intercepted the real `PUT /storyboards/:draftId` body the frontend sends on Save — `Scene 1 mediaItems: 0`, no motion_graphic item at all. The full-replace then deletes the POST-attached DB row. Verified the POST endpoint itself persists correctly (1 row) in isolation, so the drop is on the Save path.

**Root cause:** `apps/web-editor/src/features/storyboard/components/SceneModal.tsx:205` renders `<SceneModalMediaSection …>` **without an `onAttachMotionGraphic` handler**. The media section's picker calls `onAttachMotionGraphic?.(row)` (`SceneModal.mediaSection.tsx:321-322`) via optional chaining, so with the prop undefined the attached row is **never added to the modal's `mediaItems` state**. The visible "attached preview" is the picker's own internal `MotionGraphicPlayer`, not a media-list entry. At Save, `mediaItems` is empty → `handleSave` payload has no MG → autosave PUT replaces the block media with an empty set → the server row is deleted.

Note: r3's "AC-04 reload PASS" was a false positive — it observed **pre-existing** attached rows that were never re-saved through the modal, so it never exercised attach→Save→reload.

**Fix (handed to implement, TDD):** wire `onAttachMotionGraphic` in `SceneModal.tsx` to map the returned `BlockMediaMotionGraphic` row (`{ snapshot: MotionGraphicSnapshot }`) into a `ModalMediaItem` (`motionGraphic: { snapshotId: row.snapshot.id, code, durationSeconds, fps, width, height }`, `fileId:''`, appended with `sortOrder`) and pass it to `SceneModalMediaSection`. Backend schema + `replaceStoryboard` already round-trip the snapshot FK, so once the item reaches the PUT body it persists.
