Convention witnesses: `ProjectCard.tsx:99-114` (whole-card `role="button"` + Enter/Space + hover transition; inner Delete does `stopPropagation`, `:71,169`) and `AssetCard.tsx:123-141` — FlowListPage is the **only** card in the app demanding a separate Open button. Header comment at `FlowListPage.tsx:8` even claims to follow ProjectCard.

- **Fix shape (effort S):** card root gets `role="button"`, `tabIndex={0}`, `aria-label`, `onClick`/`onKeyDown` (Enter/Space) → `onOpen(flow.flowId)`, `cursor: pointer`, hover border highlight (JS hover-state per the inline-style convention); Rename (`:241`), Delete (`:263`) and the rename `<input>` row (`:165`) get `stopPropagation`; card onClick is a no-op while `isRenaming`. Remove the Open button (`:218-236`).

### U5 — Regeneration overwrites the result block instead of appending — **FIX NOW (stage-1, AC-01 violation)**

- **Sev:** MAJOR. **AC:** **AC-01 violated** — `spec.md:88` verbatim: "the system creates a new connected result block (**prior result blocks on this generation block are retained as a history of runs**)". The implementation does the opposite.
- **Finding:** `FlowEditorPage.tsx:197-207` — on `phase === 'tracking'` a result block is created **only if none exists** with that `sourceBlockId` (the `exists` guard); a second Generate reuses + visually overwrites the old block (`:307-326` resolves job/preview by `sourceBlockId`, latest-wins). Reload collapses to one job per gen block (`jobsByBlock`, `:133-140`). The backend **already preserves full history** — `getJobsByFlowId` (`aiGenerationJob.repository.ts:235-243`) returns all rows, one per run — the loss is purely client-side. Note: `spec.md:36/:190` ("exactly one result per run") concern multi-output of a *single* run — a different axis; they do not license cross-run overwrite.
- **Resolution (owner, 2026-06-04):** every run creates a NEW result block, stacked above the previous one; **failed runs' blocks are retained** in the history (deletable manually).
- **Fix shape (effort M):**
  1. `addResultBlock` (`useFlowCanvas.ts:307`) accepts + persists `params.jobId` — the per-run binding that lets N result blocks share one `sourceBlockId`. (Id scheme already collision-free: `result-<ts>-<counter>`, `useFlowCanvas.ts:309`; schema unbounded, `flowCanvas.schema.ts:85-90` — no schema change.)
  2. Drop the `exists` guard (`FlowEditorPage.tsx:197-207`) — always append on `tracking`, passing the live `jobId`; position right of the gen block, stacked **above** prior same-source results.
  3. Re-key job/preview resolution by `jobId` (`FlowEditorPage.tsx:133-140, 262-290, 307-326`): keep all rows, index by jobId; **legacy fallback** — result blocks without `params.jobId` (pre-fix flows) resolve via latest-per-sourceGen so old flows still render.
  4. Auto-connect already adds an edge per `addResultBlock` (`useFlowCanvas.ts:316-330`); verify the gen `out` handle permits multiple outgoing edges.
  5. Tests: rewrite the latest-wins e2e (`generate-ai-flow.spec.ts:430-481` — failed-then-succeeded now shows TWO result blocks, failed retained); add "Generate twice → two result blocks, oldest retained, newest above" (AC-01 history clause); re-audit `FlowEditorPage.test.tsx`, `ResultNode.render.test.tsx`; fix stale keep-latest comment `aiGenerationJob.repository.ts:236-237`.
- **Artifact amendments (part of the fix):**
  - `spec.md` AC-01 (`:88`): add the positioning + per-run-binding clause — new result block placed **above** the most recent prior result for this gen block, newest on top; each result block bound to its own job, history not collapsed on reload; failed runs retained.
  - `spec.md` retry/reattach framing + `test-plan.md`: "stale failure overridden by newer success" → "failed run's block retained alongside the success's".
  - `sad.md:236, :499-500` + `data-model.md`: document `result.params.jobId` as the run-binding discriminator; `getJobsByFlowId` returns full ordered history.
