### U1 — Selected model name unreadable on the generation node — **FIX NOW**

- **Sev:** MINOR (UX). **AC:** none violated — convention regression within the feature.
- **Finding:** when a model is selected, the model-label `<button>` leaves `color: undefined` → UA default near-black `buttontext` on the dark `SURFACE_ELEVATED` (`#1E1E2E`) node. Every sibling element sets an explicit token color; this button is the lone exception. The Inspector `<select>` itself is correctly styled (`TEXT_PRIMARY` on `SURFACE_BASE`).
- **Cites:** `apps/web-editor/src/features/generate-ai-flow/components/GenerationNode.tsx:99` (`color: model ? undefined : TEXT_SECONDARY`); `flowNodeStyles.ts:9,13` (tokens); convention witnesses `NodeDeleteButton.tsx:39`, `GenerationNode.tsx:127`, `ContentNode.tsx:97`.
- **Fix:** `color: model ? TEXT_PRIMARY : TEXT_SECONDARY` (+ optional `fontWeight: 500`). No test asserts the color → non-breaking. **Effort S.**

### U2 — TTS/STS voice selection missing: `voice_picker` field type not honored by the Inspector — **FIX NOW**

- **Sev:** MAJOR. **AC:** AC-16 / US-04 functionally violated for Speech-to-Speech (a required parameter is unreachable → the model cannot be configured from the canvas at all); implementation regression vs the committed catalog design for Text-to-Speech.
- **Finding:** the catalog declares `voice_id` as `type: 'voice_picker'` (TTS optional, STS `required: true`), but the Inspector's `ParamField` switch handles only `enum`/`boolean`/`number`/text-default — no `voice_picker` branch → TTS renders a raw free-text input for an opaque ElevenLabs voice id. Worse: `GenerationParamFields` filters out all `required` fields, so the STS `voice_id` never renders — no voice control exists for STS. A complete picker with list + per-voice audio previews already exists in the legacy single-model wizard and is backed by live endpoints (`GET /ai/voices`, `/ai/voices/available`, `/ai/voices/:voiceId/sample`).
- **Cites:** `packages/api-contracts/src/elevenlabs-models.ts:160-165` (TTS), `:234-239` (STS required); `packages/api-contracts/src/fal-models.ts:65` (`voice_picker` first-class type); `apps/web-editor/src/features/generate-ai-flow/components/Inspector.tsx:250-341` (switch — no voice_picker case), `:226-228` (required-filter kills STS voice); reuse: `apps/web-editor/src/shared/ai-generation/components/VoicePickerField.tsx`, `VoicePickerModal.tsx`, `VoicePickerRows.tsx`, hooks `useUserVoices.ts`/`useAvailableVoices.ts`; backend `apps/api/src/routes/aiGeneration.routes.ts:14,17,22`.
- **Fix:** (1) add a `voice_picker` branch in `ParamField` rendering the existing `VoicePickerField`; (2) narrow the required-filter so voice_picker fields render even when required (they are not satisfiable by a canvas connection — no modality); (3) Inspector unit test: TTS/STS block renders the picker trigger (not a text input) and a chosen voice writes `voice_id` into block params. No backend/catalog/API changes. **Effort S.**

### U3 — Cost estimate is a flat per-model constant; pricing not adjustable, ignores all price-driving params — **FIX NOW (scope escalation accepted by owner)**

- **Sev:** MAJOR (product). **AC:** no accepted AC violated — AC-11 promises only the confirmation-gate behavior. This is the documented §8 open question (spec.md:243) + ADR-0005 accepted debt, now escalated by the client into committed scope.
- **Finding:** `estimateBlockCost` reads only `block.params['modelId']` and looks it up in the compile-time `FLOW_PRICE_TABLE` — a flat USD per model. It never reads `duration`/`resolution`/`num_images`/`num_frames`/music length, although the spend path (`buildJobOptions`) forwards exactly those params to the provider. A 10s 1080p multi-image run quotes the same figure as a 5s 480p single. Price changes require a code edit + redeploy; no pricing or model-catalog table exists in the DB.
- **Cites:** `apps/api/src/lib/flow-pricing.ts:31-68`; `apps/api/src/services/flow-generate.service.ts:116-139` (estimate), `:455-497` (spend path uses the params); `apps/web-editor/.../CostConfirmModal.tsx:64,88,91-94`; price-driving catalog fields `fal-models.ts:402,480,776,816,832,947,1022,1038`, `elevenlabs-models.ts:281,293-295`; `flowCanvas.schema.ts:48` (params already persisted); `contracts/openapi.yaml:550-569` (response shape param-agnostic — no contract change needed); `spec.md:168-171` (AC-11), `:243` (§8 OQ); `adr/0005:33,42`.
- **Resolution (owner, 2026-06-04):** full scope — DB-adjustable pricing reacting to all price-driving params.
- **Fix (architecture agreed):**
  - New migration `flow_model_pricing` (model_id PK, currency, base_amount, per_second, per_image, resolution_mult JSON, updated_at), seeded from the current `FLOW_PRICE_TABLE` → day-one behavior unchanged.
  - Formula in `estimateBlockCost`: `(base + per_second × duration_s + per_image × num_images) × resolution_mult[resolution] ?? 1`; music length normalized from `duration` / `music_length_ms ÷ 1000`; round to minor unit; keep `bestEffort: true`.
  - New `flowModelPricing.repository.ts` with read-through in-process cache (live re-estimates per ADR-0005); `flow-pricing.ts` becomes DB lookup with the static map as seed + fallback.
  - Artifacts: data-model.md + new AC in spec §5 (estimate scales with duration/resolution/image-count, pricing DB-adjustable) + resolve §8 OQ at spec.md:243 + ADR amending/superseding the static-table part of 0005. **No openapi change, no frontend change.**
  - Tests: service tests asserting different duration/num_images/resolution → different amounts; pricing-repository test.
  - **Effort M.**
